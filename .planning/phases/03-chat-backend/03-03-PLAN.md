**Test cases:**

1. `POST /api/chat without auth returns 401`
   - Send POST /api/chat with no cookie
   - Assert statusCode === 401
   - Assert body.ok === false

2. `POST /api/chat with empty message returns 400`
   - Send authenticated POST /api/chat with `{ message: "" }`
   - Assert statusCode === 400
   - Assert body.error contains "message is required"

3. `POST /api/chat with missing message field returns 400`
   - Send authenticated POST /api/chat with `{}`
   - Assert statusCode === 400

4. `POST /api/chat first message creates conversation`
   - Send authenticated POST /api/chat with `{ message: "HCI + Backup for 100 users" }`
   - Assert statusCode === 201
   - Assert body.ok === true
   - Assert body.conversation_id is a string (UUID format)
   - Assert body.project_id is a string
   - Assert body.stage is a string (one of the valid stages)
   - Assert body.text is a non-empty string

5. `POST /api/chat multi-turn: first message then selection`
   - Send first message, capture conversation_id
   - Send second message: `{ conversation_id: capturedId, message: "1" }`
   - Assert statusCode === 200
   - Assert body.conversation_id === capturedId
   - Assert body.stage is 'complete' (auto-advanced through BOM + proposal)
   - Assert body.text is a non-empty string

6. `POST /api/chat with invalid conversation_id returns error`
   - Send `{ conversation_id: "00000000-0000-0000-0000-000000000000", message: "hello" }`
   - Assert response indicates error (500 or body.ok === false)

Note: These tests run in local/mock mode (no Supabase). The agents will use mock responses. The conversation CRUD will use local fallback. Verify the flow still works end-to-end in mock mode. If getConversationById returns null in local mode (no Supabase), the multi-turn test (test 5) may fail. In that case, test 5 should be marked with a comment explaining it requires Supabase, and only tests 1-4 and 6 should assert strictly.
